# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Rust REST API (axum 0.8 + tokio + sqlx 0.9 / PostgreSQL 16) for a two-vet clinic: owners, patients, visits, vaccinations, appointments, inventory, dashboard, presigned file storage, push notifications, and an offline-sync change feed for a Flutter app. OpenAPI via utoipa (Swagger UI at `/docs`).

## Commands

Local dev needs a `.env` (copy `.env.example`, set `JWT_SECRET` ≥32 chars) and a Postgres reachable via `DATABASE_URL` (README shows a throwaway `docker run` on port 5433).

```bash
make dev            # cargo run -- serve (API on :8080)
make fmt            # cargo fmt
make lint           # cargo clippy --all-targets -- -D warnings  (matches CI)
make test           # cargo test — integration tests need Docker running
make migrate        # cargo run -- migrate
make seed           # migrations + idempotent demo data (prints login creds)
make prepare        # cargo sqlx prepare — REQUIRED after changing any query/migration
make prepare-check  # verify .sqlx cache in sync (CI mode)
make deny           # cargo-deny license & advisory audit
```

Running a single test:

```bash
cargo test --lib                          # unit tests only (no Docker)
cargo test --test auth_test               # one integration test binary
cargo test --test auth_test -- refresh    # filter by test name within it
```

Integration test binaries: `auth_test`, `patients_test`, `sync_test`, `inventory_test`, `visits_test`. Each spins up a disposable Postgres 16 testcontainer.

## The sqlx offline cache (easy to forget, CI will fail)

SQL in `query!`/`query_as!` macros is checked at compile time against `DATABASE_URL`, and the results are cached in the committed `.sqlx/` directory so CI/Docker build offline (`SQLX_OFFLINE=true`). **After adding or changing any query or migration: `make prepare` and commit the updated `.sqlx/`.** CI runs `cargo sqlx prepare --check`. Integration tests deliberately use runtime-checked `sqlx::query()` so the cache only covers the library crate.

## Architecture

Everything lives in the library crate (`citra_petcare`); `src/main.rs` is CLI wiring only (`serve | migrate | seed`). Integration tests build the exact same router via `http::build_router` and drive it in-process with `tower::ServiceExt::oneshot` — no sockets (`tests/common/mod.rs` is the harness).

Each `src/domain/<module>/` follows the same four-layer shape:

- `handlers.rs` — thin HTTP layer plus a `router()` fn with utoipa annotations (routers are merged in `src/http/router.rs`, spec assembled in `src/http/openapi.rs`)
- `service.rs` — business rules
- `repo.rs` — **all SQL lives here**; rows are mapped to domain types before returning
- `models.rs` / `dto.rs` — domain types / request- and response-shapes

Layering rule (strict): handlers never touch SQL, repos never build HTTP responses, sqlx rows never cross the repo boundary.

Cross-cutting pieces:

- **`AppState`** (`src/state.rs`): `PgPool`, `Arc<Config>`, `Arc<dyn Storage>`, `Arc<dyn Notifier>`. The trait objects (async_trait) let dev/prod/tests swap drivers: storage = S3 (OCI Object Storage, presigned PUT/GET) or local disk with HMAC-signed URLs; notifier = FCM HTTP v1 or a log driver.
- **Errors**: domain `thiserror` enums convert via `#[from]` into the central `AppError` (`src/error.rs`), whose single `IntoResponse` renders RFC 7807 `application/problem+json` with stable `type` URNs. `sqlx::Error` is classified there (unique/FK/check violations → 409/422). Repo/service code just uses `?`. `anyhow` is by convention only used in `main()` and tests.
- **Auth**: adding the `AuthUser` extractor (`domain/auth/extractor.rs`) as a handler parameter *is* the auth guard. JWT access tokens (HS256, 15 min) + rotating refresh tokens (SHA-256-hashed at rest, reuse detection revokes the session family). No public registration — users are seeded.
- **Validation**: `ValidatedJson` extractor (`src/http/extract.rs`) wires the validator crate into 422 problem+json with a per-field `errors` map.
- **Migrations** are embedded in the binary (`src/db.rs`); `serve` applies them on boot when `AUTO_MIGRATE=true` (default).
- **Config** (`src/config.rs`): typed env config; boot fails fast listing *every* missing/invalid variable at once.
- **Inventory**: current stock is never stored — it's `SUM(±qty)` over the stock-movement ledger; OUT movements row-lock the item to prevent overselling.

## API & data conventions

- Base path `/api/v1`; JSON is camelCase; enums are SCREAMING_SNAKE (`CAT`, `NO_SHOW`).
- Cursor pagination: `?cursor=&limit=` → `{ data, meta: { limit, nextCursor, hasMore } }` (`src/http/pagination.rs`).
- Offline-sync contract: UUIDv7 primary keys may be generated by the client; `PUT /{entity}/{id}` is an idempotent full-representation upsert (upserting a soft-deleted row resurrects it); every table has `created_at`/`updated_at`/`deleted_at` (soft delete), with `updated_at` bumped by a DB trigger as the sync cursor for `GET /sync/changes?since=`.
- Every error path returns RFC 7807 — keep new errors in that pipeline (new domain error variant → `AppError` mapping), don't hand-roll status responses.

## CI

`.github/workflows/ci.yml`: `cargo fmt --check`, clippy with `-D warnings`, `cargo sqlx prepare --check`, `cargo test` (runner has Docker), cargo-deny, Docker image build — then an automatic deploy to the production VPS on every push to `main` via a self-hosted runner. Deployment/ops details are in the README.

`unsafe_code` is forbidden crate-wide (Cargo.toml lint).
